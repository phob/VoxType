# Contributing To VoxType

## Commits And Releases

VoxType uses Conventional Commits with Release Please.

Use this commit format:

```text
type(scope): short imperative summary
```

Examples:

```text
feat(dictation): add push-to-talk recording
fix(insertion): restore clipboard after paste fails
perf(asr): reduce whisper worker startup time
docs(planning): clarify OCR dictionary strategy
ci(release): add release-please workflow
```

Release-relevant commit types:

- `feat`: user-visible feature, usually a minor version bump.
- `fix`: user-visible bug fix, usually a patch version bump.
- `perf`: user-visible performance improvement, usually a patch version bump.
- `feat!` or `BREAKING CHANGE:`: breaking change, major version bump after 1.0.

Internal-only types:

- `docs`
- `refactor`
- `test`
- `build`
- `ci`
- `chore`

Write release-relevant commit messages for users, not as implementation notes. Release Please will use these commits to open a release PR that updates `CHANGELOG.md`, bumps `package.json`, updates `.release-please-manifest.json`, and creates a GitHub Release when merged.

## Changelog Files

- `CHANGELOG.md` is the public release changelog generated by Release Please.
- `planning/changelog.md` is only the internal planning-memory changelog.
